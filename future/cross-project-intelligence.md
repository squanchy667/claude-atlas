# Future Feature: Cross-Project Intelligence

> Planned for v2. Personal intelligence could ship earlier. Collective layer requires marketplace infrastructure.

## Overview

Cross-Project Intelligence is the knowledge layer that makes every Atlas project smarter than the last. When a project completes, it generates an intelligence report containing anonymized patterns, calibrated estimates, proven decisions, and common mistakes. This intelligence feeds forward --- into your next project (personal intelligence) and into every Atlas user's next project (collective intelligence).

The more projects completed through Atlas, the smarter the system gets. This is not a feature that can be cloned by copying the code. It is a data moat that compounds with every project.

## Three Layers

### Layer 1: Personal Intelligence

Your own project history, stored locally and used to calibrate your future projects.

**What it contains:**
- Completed project summaries (from `/atlas-retro`)
- Your estimation accuracy per task type (auth tasks, API tasks, UI tasks, etc.)
- Your recurring mistakes and how you resolved them
- Your most-used skills and patterns
- Your typical phase duration by project type
- Your drift patterns (where you tend to deviate)

**Where it lives:**
- `~/.atlas/intelligence/personal/` (local to your machine, never shared without consent)
- One JSON file per completed project: `project-slug-report.json`
- One aggregated profile: `calibration.json` (updated after each project)

**How it is used:**
- `/atlas-setup` reads `calibration.json` to pre-calibrate estimates
- `/plan-phase` adjusts complexity estimates based on your history
- `/task plan` surfaces relevant mistakes from past projects: "In your last 3 projects, auth middleware tasks drifted 60% of the time. Common cause: token refresh strategy not decided upfront."
- Memory files from previous similar projects can be optionally imported

### Layer 2: Collective Intelligence

Anonymized, aggregated data from all Atlas projects worldwide. Stored in the marketplace repository and updated periodically.

**What it contains:**
- Aggregated estimation accuracy by project type and task category
- Most common mistakes by project type (anonymized)
- Most effective patterns by project type
- Most-used skills by project type
- Typical phase structure and duration by project type
- Drift frequency by task category
- First-try approval rates by task category

**Where it lives:**
- `marketplace/intelligence/` in the marketplace repository
- Organized by project type: `saas/`, `api/`, `cli/`, `game/`, `extension/`, `library/`
- Each project type directory contains aggregated JSON files

### Layer 3: Project Intelligence Report

Generated by `/atlas-retro` at the end of each project. Contains the full analysis of that specific project, from which personal and collective intelligence are derived.

**What it contains:**
- Project summary: type, stack, phases, total tasks, duration
- Estimation accuracy: per-task predicted vs actual complexity
- Quality scorecard (see quality-scorecard.md)
- Mistake catalog: what went wrong, root cause, resolution
- Pattern catalog: what worked well, why, applicability
- Decision review: each major decision, outcome, whether it was the right call
- Drift analysis: frequency, severity, root causes, systemic patterns
- Skill effectiveness: which skills were used, which were most valuable, which were unused
- Phase-by-phase breakdown: duration, task count, drift rate, first-try approval rate

## The `/atlas-retro` Command

This is the primary command for generating project intelligence.

### Invocation

```
/atlas-retro
```

Run after the final phase is complete. Reads the entire Atlas docs repository and produces a comprehensive retrospective.

### Step-by-Step Process

**Step 1: Gather Data**

Read all project files:
- All `tasks/phase-N/task-N/outcome.md` files (what was built)
- All `tasks/phase-N/task-N/drift.md` files (what deviated)
- All `tasks/phase-N/task-N/agreement.md` files (what was planned)
- All `checkpoints/passed/` files (what was approved, how many tries)
- All `memory/` files (decisions, mistakes, patterns, drift-log)
- All `skills/` files (what skills were used, created, archived)
- `status/progress.md` (final status)
- `status/blockers.md` (blocker history)

**Step 2: Compute Aggregates**

Calculate project-wide metrics:
- Total tasks, phases, and duration
- Tasks by status (done, blocked, deferred)
- Drift count and severity distribution
- Checkpoint revision count (how many checkpoints needed revisions before approval)
- Skill creation vs import ratio
- Blocker count and average resolution time

**Step 3: Analyze Estimation Accuracy**

For each task, compare:
- Agreement's estimated complexity vs outcome's actual indicators
- Planned scope vs actual scope (from drift records)
- Group by task category (auth, database, API, UI, integration, infrastructure)
- Calculate per-category estimation accuracy percentage

**Step 4: Generate Mistake Summary**

From `memory/mistakes.md` and drift records:
- Categorize mistakes by type (architectural, implementation, planning, communication)
- Identify root causes
- Note which mistakes were repeated vs one-time
- Anonymize for collective sharing (remove project names, specific code, proprietary details)

**Step 5: Generate Pattern Summary**

From `memory/patterns.md` and successful outcomes:
- Identify patterns that worked well
- Note patterns that were adopted from skills vs discovered during the project
- Rate pattern impact (high/medium/low based on how many tasks benefited)

**Step 6: Review Decisions**

From `memory/decisions.md`:
- List each major decision with date and rationale
- Assess outcome: did the decision hold, was it revised, did it cause downstream issues?
- Categorize: good call / acceptable trade-off / should have decided differently

**Step 7: Compute Quality Scorecard**

Calculate the full quality scorecard (see quality-scorecard.md for metric definitions):
- First-try approval rate
- Drift rate and severity
- Estimation accuracy
- Blocker resolution time
- Skill reuse rate

**Step 8: Generate Report**

Write the Project Intelligence Report to `retro/project-report.md`:

```markdown
# Project Intelligence Report: [Project Name]

## Summary
- Type: [type]
- Stack: [stack]
- Phases: [N] completed
- Tasks: [N] total ([N] done, [N] deferred, [N] blocked)
- Duration: [sessions/time]
- Overall Health: [GOOD/FAIR/NEEDS_IMPROVEMENT]

## Quality Scorecard
[Full scorecard — see quality-scorecard.md]

## Estimation Accuracy
[Per-category accuracy table]

## What Went Well
[Pattern summary — things that worked]

## What Went Wrong
[Mistake summary — categorized, with root causes]

## Decisions Review
[Decision-by-decision assessment]

## Drift Analysis
[Systemic drift patterns, root causes, frequency]

## Skill Effectiveness
[Which skills were valuable, which were unused, which were missing]

## Recommendations for Next Project
[Concrete suggestions based on this project's data]
```

**Step 9: Offer Publishing**

Ask the user: "Would you like to publish anonymized intelligence to the collective? This helps calibrate estimates and surface common patterns for all Atlas users. No project names, company names, or proprietary code is shared."

If yes:
1. Strip all identifying information (project name, company, specific file paths, code snippets)
2. Keep only: project type, task categories, estimation data, mistake categories, pattern categories, quality metrics, skill usage
3. Package as `intelligence-report-[hash].json`
4. Add to `marketplace/intelligence/[project-type]/`
5. Commit with message: "Add anonymized intelligence report for [project-type] project"

**Step 10: Update Personal Intelligence**

Update `~/.atlas/intelligence/personal/calibration.json` with this project's data:
- Merge estimation accuracy into running averages
- Add new mistakes to personal mistake catalog
- Add new patterns to personal pattern catalog
- Update skill usage statistics

## How New Projects Use Intelligence

### During `/atlas-setup`

When a new project is created:

1. Check for personal intelligence at `~/.atlas/intelligence/personal/calibration.json`
2. Check for collective intelligence at `marketplace/intelligence/[project-type]/`
3. If either exists, present a summary:

```
Based on your 4 completed SaaS projects:
- Auth tasks take 40% longer than you estimate
- Database tasks are well-calibrated
- You tend to defer error handling — recommend including in Phase 1

Based on 127 completed SaaS projects from the Atlas community:
- Most common mistake: deferred token refresh strategy (65% of projects)
- Most effective pattern: validate data model with seed data before features
- Typical scope: 35 tasks across 5 phases
- Most valuable skills: auth-patterns, api-design, error-handling
```

4. Pre-load relevant intelligence into `atlas/gaps.md` and `memory/mistakes.md`

### During `/plan-phase`

When planning a phase:

1. Read estimation calibration data for the task types in this phase
2. Adjust complexity estimates: "You estimated T005 (auth middleware) as simple. Historical accuracy for auth tasks at simple complexity: 58%. Recommend moderate."
3. Surface relevant mistakes: "In 65% of SaaS projects, auth middleware tasks drift because token refresh was not decided. Has this been decided? Check atlas/gaps.md."
4. Suggest skills: "Auth-patterns skill was rated 4.5/5 across 80 projects. Import it?"

### During `/task plan`

When planning a specific task:

1. Check personal history for similar task types
2. Surface relevant mistakes and patterns
3. Adjust estimated complexity if calibration data suggests it

## The Data Moat

Skills can be copied --- they are markdown files. Templates can be cloned --- they are directory structures. But collective intelligence from hundreds of real completed projects cannot be replicated. It requires:

- Real projects being completed through Atlas
- Real estimation data being generated
- Real mistakes being documented
- Real patterns being validated

Every project that completes through Atlas makes the intelligence layer more valuable. A competitor would need to bootstrap from zero. This is the core long-term moat.

## Privacy

Intelligence reports are anonymized through a multi-step process before collective sharing:

1. **Strip identifiers:** Remove project name, company name, author name, repository URLs
2. **Strip code:** Remove all code snippets, file paths, specific error messages
3. **Generalize descriptions:** "Fixed Stripe webhook signature validation" becomes "Fixed payment webhook authentication"
4. **Aggregate:** Individual task data is only shared as category-level statistics, never as individual records
5. **User consent:** Publishing is always opt-in, prompted during `/atlas-retro`, never automatic

Users can review the anonymized report before publishing. They can also redact specific entries they consider sensitive.

## Status

- **Personal intelligence** could ship as early as v1.5. It only requires local storage and `/atlas-retro`.
- **Collective intelligence** requires marketplace infrastructure and a critical mass of projects.
- **`/atlas-retro`** as a personal-only tool is a good candidate for early release.
