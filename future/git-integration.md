# Future Feature: Git Integration

> Planned for v2. Individual features (auto-branching, auto-PR) could ship incrementally.

## Overview

Git Integration automates the git workflows that Atlas tasks naturally produce. Instead of manually creating branches, writing PR descriptions, tagging releases, and generating changelogs, Atlas commands handle these as part of the normal task and phase lifecycle.

This feature uses `git` and `gh` (GitHub CLI) --- both free, both standard, both already installed on most developer machines. No custom infrastructure, no hosted services, no API keys beyond what `gh` already manages.

## Features

### Auto-Branching

**Trigger:** `/task start` (or equivalent when beginning work on a task)

**What happens:**
1. Atlas reads the current task's ID and description from `tasks/phase-N/task-N/task.md`
2. Creates a branch from the current base branch (typically `main`):
   ```
   atlas/phase-2-task-5-user-authentication
   ```
3. Switches to the new branch
4. Records the branch name in `status/progress.md` alongside the task status

**Branch naming convention:**
```
atlas/phase-{N}-task-{M}-{description-slug}
```
- `atlas/` prefix identifies Atlas-managed branches
- Phase and task numbers provide ordering and traceability
- Description slug is derived from the task title (lowercase, hyphens, max 40 chars)

**Behavior details:**
- If the branch already exists (task was paused and resumed), Atlas switches to it without creating a new one
- If the base branch has advanced since the branch was created, Atlas warns and offers to rebase
- Branch creation is logged in `status/progress.md`

**Configuration:** The base branch name is configurable in `project/overview.md` (defaults to `main`). Some projects use `develop` or a release branch.

### Auto-PRs

**Trigger:** `/task complete` (when a task's checkpoint is ready for review)

**What happens:**
1. Atlas pushes the current branch to the remote: `git push -u origin atlas/phase-2-task-5-user-authentication`
2. Creates a draft PR via `gh pr create`:
   ```
   gh pr create \
     --title "Phase 2 Task 5: User Authentication" \
     --body "[generated body]" \
     --draft \
     --base main
   ```
3. Records the PR URL in `status/progress.md` and in the task's `outcome.md`

**PR body template:**

```markdown
## Task
Phase 2, Task 5: User Authentication

## Agreement Summary
[Extracted from tasks/phase-2/task-5/agreement.md — scope and acceptance criteria]

## What Was Built
[Extracted from tasks/phase-2/task-5/outcome.md — actual implementation summary]

## Drift Events
[If any drift occurred, listed from tasks/phase-2/task-5/drift.md]
[If no drift: "No drift events."]

## Self-Validation
[Extracted from the checkpoint file — Claude's honest assessment of completeness]

## Files Changed
[List of files added, modified, deleted — from git diff]

## Test Results
[Test pass/fail summary if tests were run]

---
*Generated by Atlas. Full task context: `tasks/phase-2/task-5/`*
```

**PR lifecycle:**
- Created as **draft** when `/task complete` runs
- Human reviews the checkpoint in Atlas AND the PR in GitHub
- When the checkpoint moves to `passed/`, Atlas marks the PR as ready for review: `gh pr ready`
- When the human merges the PR (manually or via `/checkpoint approve`), Atlas records the merge commit

**Why draft PRs:** The PR should not be reviewed before the Atlas checkpoint is reviewed. The draft status signals "this is ready for Atlas review, not yet ready for code review." This prevents premature code review before the agreement-level review is complete.

### Auto-Releases

**Trigger:** `/complete-phase` (when a phase is fully complete and all checkpoints are passed)

**What happens:**
1. Atlas creates a git tag on the current commit:
   ```
   git tag -a v0.2.0-phase-2 -m "Phase 2: Core Features complete"
   ```
2. Generates release notes from all outcome.md files in the phase
3. Creates a GitHub release via `gh release create`:
   ```
   gh release create v0.2.0-phase-2 \
     --title "Phase 2: Core Features" \
     --notes "[generated notes]" \
     --prerelease
   ```

**Release notes template:**

```markdown
# Phase 2: Core Features

## Completed Tasks

### Task 5: User Authentication
[Brief summary from outcome.md]
- Files: [count] added, [count] modified
- Drift: [count] events ([severity])

### Task 6: Role-Based Access Control
[Brief summary from outcome.md]
- Files: [count] added, [count] modified
- Drift: None

### Task 7: API Endpoint Layer
[Brief summary from outcome.md]
- Files: [count] added, [count] modified
- Drift: [count] events ([severity])

## Phase Quality
[Phase scorecard from quality-scorecard.md metrics]

## Known Issues
[From status/blockers.md — any unresolved items carried forward]
```

**Version numbering:**
- `v0.N.0-phase-N` for pre-release phase completions
- `v1.0.0` for the first production release (user-triggered, not automatic)
- Minor version increments with each phase
- The version scheme is configurable in `project/overview.md`

**Prerelease flag:** Phase releases are always marked as `--prerelease` in GitHub. Only a deliberate production release (manually triggered by the user) creates a non-prerelease.

### Auto-Changelog

**Trigger:** `/changelog [scope]` (explicit command invocation)

**Scope options:**
- `/changelog phase-2` --- changelog for a single phase
- `/changelog v0.1.0..v0.3.0` --- changelog for a version range
- `/changelog all` --- full project changelog

**What happens:**
1. Atlas reads all `outcome.md` files within the specified scope
2. Categorizes each outcome:
   - **Added:** New features or capabilities
   - **Changed:** Modifications to existing behavior
   - **Fixed:** Bug fixes
   - **Removed:** Deprecated or removed functionality
   - **Infrastructure:** CI/CD, tooling, configuration changes
3. Generates or updates `CHANGELOG.md` at the repository root

**Changelog format (Keep a Changelog convention):**

```markdown
# Changelog

## [0.3.0-phase-3] - 2026-03-20

### Added
- Payment processing via Stripe integration (T015)
- Email notifications for account events (T016)
- Webhook handler for external service events (T017)

### Changed
- Database connection pooling increased to handle payment webhooks (T015)

### Fixed
- Auth token refresh race condition during concurrent requests (T014)

## [0.2.0-phase-2] - 2026-03-10

### Added
- User authentication with JWT tokens (T005)
- Role-based access control with 3 roles (T006)
- REST API endpoint layer with validation (T007)
- Admin dashboard with user management (T008)

### Infrastructure
- Database migration framework configured (T009)
```

**Linking:** Each entry includes the task ID in parentheses. If auto-PRs are enabled, the task ID links to the corresponding PR.

## Configuration

Git integration settings live in `project/overview.md` under a dedicated section:

```markdown
## Git Integration

- **Base branch:** main
- **Auto-branching:** enabled
- **Auto-PRs:** enabled (draft)
- **Auto-releases:** enabled (prerelease)
- **Auto-changelog:** enabled
- **Version scheme:** semver-phase (v0.N.0-phase-N)
- **Remote:** origin
```

Each feature can be independently enabled or disabled. Users who prefer manual git workflows can disable everything and use Atlas purely for planning and tracking.

## Merge Strategy

Atlas does not enforce a merge strategy. It creates branches and PRs --- the merge is handled by the human through their normal workflow (merge commit, squash, rebase --- whatever the team prefers).

However, Atlas records the merge method used for each task's PR. This data feeds into the retrospective for teams that want to standardize.

## Conflict Handling

When a task branch has conflicts with the base branch:

1. Atlas detects the conflict when attempting to create a PR or when `/task complete` runs
2. Reports the conflicting files
3. Does NOT attempt automatic resolution --- conflicts are flagged as blockers in `status/blockers.md`
4. The human resolves conflicts manually
5. After resolution, `/task complete` can be re-run

In team mode, conflict detection is enhanced with the collision detection system (see team-mode.md). Atlas can predict likely conflicts before they happen based on which developers are touching which files.

## Multi-Repo Support

Some Atlas projects span multiple repositories (e.g., a separate frontend and backend repo). Git integration handles this by:

1. Tracking the active repository in the task context (task.md specifies which repo it targets)
2. Creating branches in the correct repository
3. Creating PRs in the correct repository
4. Cross-linking PRs when tasks span repositories (PR body includes links to related PRs in other repos)

This is configured in `project/architecture.md` under a "Repositories" section that maps logical components to repository paths.

## Tools Required

| Tool | Required | Purpose |
|------|----------|---------|
| `git` | Yes | Branch management, tagging, committing |
| `gh` | Yes (for PR/release features) | PR creation, release creation, PR status updates |

Both are free, open-source CLI tools. `gh` requires a one-time authentication (`gh auth login`). Atlas checks for their presence at `/atlas-setup` time and warns if either is missing.

## Connection to Other Features

### Quality Scorecard
PR approval time and merge-to-release cycle time could become additional quality metrics in future versions.

### Team Mode
Auto-branching integrates with team mode's collision detection. When two developers' task branches touch the same files, the collision is detected early and a sync point is recommended.

### Cross-Project Intelligence
Git metrics (PR count, merge frequency, conflict rate) become part of the project intelligence report, providing collective data on how team size and project type affect git workflow patterns.

1. **Auto-branching** (v1.5) --- lowest complexity, highest immediate value
2. **Auto-changelog** (v1.5) --- standalone command, no lifecycle integration needed
3. **Auto-PRs** (v2) --- requires lifecycle integration with `/task complete` and checkpoint approval
