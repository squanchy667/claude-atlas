# Future Feature: Quality Scorecard

> Planned for v2. Some metrics could ship earlier as `/status` enhancements.

## Overview

The Quality Scorecard is a computed, quantitative measure of project health at the task, phase, and project level. It replaces gut feelings about "how is the project going?" with concrete, data-driven metrics derived from Atlas's own tracking files. The scorecard is displayed by `/status --health`, included in phase completion summaries, and forms a core component of the Project Intelligence Report generated by `/atlas-retro`.

The scorecard does not judge. A low first-try approval rate does not mean the developer is bad --- it might mean the requirements were ambiguous, the problem space was unfamiliar, or the project is genuinely complex. The scorecard surfaces patterns. The human interprets them.

## Metrics

### First-Try Approval Rate

**Scope:** Task, Phase, Project

**What it measures:** The percentage of checkpoints that were approved by the human on the first submission, without revisions.

**How it is calculated:** For each checkpoint in `checkpoints/passed/`, count how many went directly from pending to passed without being returned for revisions. A checkpoint that was returned, revised, and then passed counts as "not first-try."

**How to detect revisions:** Each checkpoint file includes a `Revision` field. If `Revision: 0`, it was first-try. If `Revision: 1+`, it required revisions.

**Thresholds:**
- Good: >75%
- Fair: 50-75%
- Needs improvement: <50%

**What it indicates:** High first-try approval suggests clear agreements, accurate implementation, and honest self-validation. Low rates may indicate ambiguous agreements, implementation gaps, or overly optimistic self-assessment.

---

### Drift Rate

**Scope:** Task, Phase, Project

**What it measures:** The percentage of tasks that have at least one entry in their `drift.md` file.

**How it is calculated:** Count tasks with non-empty `drift.md` files divided by total tasks completed.

**Thresholds:**
- Good: <25%
- Normal: 25-40%
- High: >40%

**What it indicates:** Some drift is normal and expected --- requirements clarify during implementation. A high drift rate may indicate insufficient planning, unclear requirements, or a rapidly evolving problem space.

---

### Average Drift Severity

**Scope:** Phase, Project

**What it measures:** The distribution of drift severity across all drift entries.

**How it is calculated:** Each drift entry has a severity (Low, Medium, High, Critical). Count the frequency of each severity level. The average is the median severity.

**Thresholds:**
- Good: Mostly Low, some Medium
- Fair: Mixed Low/Medium with occasional High
- Needs improvement: Frequent High or any Critical

**What it indicates:** Low-severity drift (minor implementation adjustments) is healthy. High-severity drift (scope changes, architectural pivots) suggests planning gaps or requirements instability.

---

### Test Pass Rate

**Scope:** Task

**What it measures:** The percentage of tests passing at the time the task's checkpoint is submitted.

**How it is calculated:** Read from the checkpoint file's self-validation section, which reports test results. If no tests exist for the task, this metric is marked N/A.

**Thresholds:**
- Good: 100%
- Fair: 90-99%
- Needs improvement: <90%
- Integrity concern: Failing tests that were modified or skipped (see integrity rules)

**What it indicates:** Test pass rate should be 100% at checkpoint submission. Anything less means the task is not complete per its own definition. This metric is primarily useful for catching patterns where tests are habitually incomplete at submission.

---

### Estimation Accuracy

**Scope:** Task, Phase

**What it measures:** How well the estimated complexity (from `agreement.md`) matched the actual effort (from `outcome.md` indicators).

**How it is calculated:**

Each agreement includes an estimated complexity: simple, moderate, or complex. Each outcome records actual effort indicators:
- Files changed (count)
- Drift entries (count)
- Revisions (count)
- Qualitative complexity note from Claude

The estimation accuracy model maps these indicators to an actual complexity level:
- **Simple actual:** 1-3 files changed, 0 drift entries, 0 revisions
- **Moderate actual:** 4-8 files changed, 0-1 drift entries, 0-1 revisions
- **Complex actual:** 9+ files changed, 2+ drift entries, or 2+ revisions

Accuracy = percentage of tasks where estimated complexity matched actual complexity (within one level).

**Thresholds:**
- Good: >80%
- Fair: 60-80%
- Needs improvement: <60%

**What it indicates:** Estimation accuracy improves with experience. Low accuracy is normal for unfamiliar problem spaces. Systematically underestimating a specific task type (e.g., always underestimating auth tasks) is actionable insight that feeds into adaptive estimation.

---

### Blocker Resolution Time

**Scope:** Project

**What it measures:** The average duration that blockers in `status/blockers.md` remain open before resolution.

**How it is calculated:** Each blocker entry includes a creation date. Resolved blockers include a resolution date. The metric is the average difference across all resolved blockers.

**Thresholds:**
- Good: <1 session
- Fair: 1-2 sessions
- Slow: >2 sessions

**What it indicates:** Long-lived blockers suggest external dependencies, unclear requirements, or decisions that need human input but are not getting it. A pattern of slow blocker resolution is a process issue, not a technical one.

---

### Skill Reuse Rate

**Scope:** Phase, Project

**What it measures:** The ratio of imported/existing skills used during a phase vs new skills created during that phase.

**How it is calculated:** Count skills referenced in task outcomes that existed before the phase started (imported or foundation) vs skills created during the phase. Reuse rate = existing skills used / (existing skills used + new skills created).

**Thresholds:**
- Good: >50% reuse
- Fair: 30-50% reuse
- Low: <30% reuse

**What it indicates:** High reuse means the skill library is well-curated and applicable. Low reuse might mean the project is in unfamiliar territory (expected for novel projects) or that existing skills need updating. This metric feeds into skill marketplace recommendations.

---

### Agreement Revision Rate

**Scope:** Project

**What it measures:** The percentage of agreements that were revised between initial draft and approval.

**How it is calculated:** Each agreement file tracks its revision history. Count agreements with revisions vs total agreements.

**Thresholds:**
- Normal: 10-30% (some refinement is healthy)
- High: >40% (agreements may be too vague initially)
- Suspicious: 0% (are agreements being rubber-stamped?)

**What it indicates:** Some agreement revision is healthy --- it means the human is reading and refining the plan. No revisions at all might mean agreements are not being reviewed carefully. Very high revision rates might mean initial plans need more thought.

## Health Indicator Display

When `/status --health` is run, the scorecard is displayed as a tree:

```
Project Health: GOOD
├── First-try approvals: 85% (good: >75%)
├── Drift rate: 30% (normal: <40%)
├── Avg drift severity: Low (good: mostly Low)
├── Estimation accuracy: 78% (fair: >80% is good)
├── Open blockers: 0 (good: 0)
├── Skill reuse: 60% (good: >50%)
└── Agreement revisions: 20% (normal: 10-30%)
```

The overall health indicator (GOOD, FAIR, NEEDS_IMPROVEMENT) is computed as a weighted composite:

| Metric | Weight | Rationale |
|--------|--------|-----------|
| First-try approval rate | 0.25 | Strongest signal of implementation quality |
| Drift rate + severity | 0.20 | Combined planning and execution quality |
| Estimation accuracy | 0.15 | Calibration quality (improves over time) |
| Test pass rate | 0.15 | Code quality at submission |
| Blocker resolution time | 0.10 | Process health |
| Skill reuse rate | 0.10 | Knowledge management quality |
| Agreement revision rate | 0.05 | Planning collaboration quality |

**Composite score mapping:**
- GOOD: weighted score > 0.75
- FAIR: weighted score 0.50-0.75
- NEEDS_IMPROVEMENT: weighted score < 0.50

## Integration Points

### `/status --health`

Shows the current scorecard at any point during the project. Calculates metrics from all completed tasks up to the current point. Useful for mid-project health checks.

### `/complete-phase`

When a phase is completed, the phase-level scorecard is included in the phase completion summary. This gives the human a quantitative view of how the phase went before deciding whether to proceed.

```markdown
## Phase 2 Scorecard
- First-try approvals: 8/10 (80%)
- Drift rate: 2/10 tasks (20%)
- Drift severity: 1 Low, 1 Medium
- Estimation accuracy: 7/10 correct (70%)
- Blockers resolved: 1 (resolved in <1 session)
- Skills reused: 4 existing, 2 new (67% reuse)
```

### `/atlas-retro`

The full project scorecard is a core component of the Project Intelligence Report. It provides the quantitative foundation for the retrospective analysis. When anonymized and published to collective intelligence, the scorecard data feeds into cross-project benchmarking.

### Cross-Project Benchmarking

When collective intelligence has enough data, the scorecard gains a comparative dimension:

```
Project Health: GOOD
├── First-try approvals: 85% (good: >75%) — vs 72% avg for SaaS projects
├── Drift rate: 30% (normal: <40%) — vs 35% avg for SaaS projects
├── Estimation accuracy: 78% (fair: >80%) — vs 68% avg for SaaS projects
```

This tells the developer not just how their project is doing in absolute terms, but relative to similar projects. "Your estimation accuracy is better than average" is more useful than "your estimation accuracy is 78%."

## Data Collection

No new data collection is required. Every metric in the scorecard is derived from files that Atlas already creates and maintains:

| Metric | Source File(s) |
|--------|---------------|
| First-try approval rate | `checkpoints/passed/*.md` (revision count) |
| Drift rate | `tasks/phase-N/task-N/drift.md` (existence and content) |
| Drift severity | `tasks/phase-N/task-N/drift.md` (severity field) |
| Test pass rate | `checkpoints/passed/*.md` (self-validation section) |
| Estimation accuracy | `agreement.md` (estimated) vs `outcome.md` (actual) |
| Blocker resolution time | `status/blockers.md` (created/resolved dates) |
| Skill reuse rate | `skills/INDEX.md` + `outcome.md` references |
| Agreement revision rate | `agreement.md` (revision history) |

This is a pure read-and-compute feature. It adds no new files to the Atlas structure and requires no additional effort from the developer or Claude.
